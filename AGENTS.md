## Git 提交规范

### 提交时机

| 场景 | 说明 |
|------|------|
| 完成一个独立功能 | 功能可运行、无明显报错 |
| 修复一个 bug | 验证修复有效 |
| 重构完成 | 行为不变，测试通过 |
| 切换上下文前 | 避免丢失工作进度 |
| 代码审查前 | 便于 Review |

### 不要提交的情况

- 代码编译不过
- 功能未完成且无法独立运行
- 临时调试代码未清理
- 大量 `console.log` / `dbg!` 未删除

### Commit Message 格式

```
<type>: <简短描述>

<可选：详细说明>
```

类型：`feat` / `fix` / `refactor` / `docs` / `style` / `test` / `chore`

### 远程提交规则

- **自动提交本地**：按提交时机规范自动执行 `git commit`
- **禁止自动推送**：未经用户明确允许，不得执行 `git push`
- **远程操作需授权**：`push`、`create PR`、`merge` 等远程操作必须用户确认

## 提交前文档维护规则

在执行本地 `git commit` 前，Agent 必须根据本次变更内容检查并维护 4 级文档体系。

### 触发时机

- 每次准备执行本地 `git commit` 前必须执行。
- 无代码变更、仅临时调试且不提交时不触发。
- 远程操作前不单独触发；远程操作必须基于已完成的本地提交。

### 维护层级

| 变更类型 | 必须检查/维护的文档 |
|----------|--------------------|
| 产品定位、核心功能、路线图、设计取舍变化 | `PLAN.md` |
| 架构、模块职责、数据流、协议、数据库、运行模式变化 | `WHITEPAPER.md` |
| 单个功能开发、功能修复、阶段性开发进度 | `docs/{feature-name}.md` |
| 具体实现、注释、测试、脚本 | 对应源码文件 |

### 执行规则

1. Agent 在提交前必须先判断本次变更影响哪个层级。
2. 若影响 `PLAN.md`，只更新战略层内容，不写实现细节。
3. 若影响 `WHITEPAPER.md`，只更新技术规范、架构、数据流、协议、数据库等内容。
4. 若是功能开发或修复，必须维护对应 `docs/{feature-name}.md`，包括开发进度。
5. 若只是代码实现细节变化，优先更新代码注释或测试，不强行更新上层文档。
6. 文档无须更新时，Agent 必须在提交前说明："本次变更不影响文档层级，无需更新文档。"

### 提交前检查顺序

1. 检查代码变更内容。
2. 判断影响层级。
3. 更新对应文档。
4. 运行必要验证。
5. 递增版本号并同步版本文件。
6. 执行本地 `git commit`。
7. 禁止自动 `git push`。

## 项目架构速查

- 前端：React + TypeScript + TanStack Query + i18next
- 后端：Tauri (Rust) + SQLite
- 通信：Tauri `invoke` 命令，Rust 端 `#[tauri::command]` 函数

## 文档体系规范

项目文档分为4级，层级关系如下：

```
PLAN.md (战略层)
    ↓ 扩展
WHITEPAPER.md (规范层)
    ↓ 细化
docs/*.md (开发文档层)
    ↓ 实现
src/**/*.rs / src/**/*.tsx (代码层)
```

### Level 1: PLAN.md — 战略规划层

| 职责 | 内容 |
|------|------|
| 项目愿景 | 产品定位、核心价值主张、目标用户 |
| 功能总览 | 已实现特性清单、功能矩阵 |
| 技术架构概览 | 技术栈选型、整体架构图（非细节） |
| 开发路线图 | P0/P1/P2/P3 优先级划分、待开发项 |
| 设计取舍 | 关键技术决策及其理由 |

**维护原则**：只写"做什么"和"为什么"，不写"怎么做"

### Level 2: WHITEPAPER.md — 技术规范层

| 职责 | 内容 |
|------|------|
| 系统架构详解 | 分层架构、模块职责、数据流向 |
| 协议规范 | 5种协议适配器规范、转换规则、边界 |
| 数据模型 | 表结构设计、字段语义、关系说明 |
| 运行模式 | Desktop/Web/Headless 模式差异 |
| 技术边界 | 已知限制、降级策略、兼容矩阵 |
| 设计规范 | 数据库迁移规范、错误处理规范、日志规范 |

**维护原则**：写"怎么做"的规范和约束，是实现的蓝图

### Level 3: docs/*.md — 功能开发文档层

**一个功能一个文件**，文件名格式：`docs/{feature-name}.md`

| 章节 | 内容 |
|------|------|
| 概述 | 功能目标、背景、关联模块 |
| 需求分析 | 功能拆解、边界识别、风险评估 |
| 实现计划 | 任务分解、文件清单、依赖关系 |
| 开发进度 | 任务清单 + 状态标记（⏳待开始 / 🔄进行中 / ✅已完成 / ❌阻塞） |
| 技术决策 | 方案对比、最终选择及理由 |
| 问题记录 | 遇到的问题、解决方案、经验教训 |
| 验收标准 | 完成条件、测试要点 |

**维护原则**：一功能一文件，开发过程中持续更新进度状态，完成后保留归档

### Level 4: 编码文件 — 实现层

| 职责 | 内容 |
|------|------|
| 源代码 | Rust 后端、React 前端、配置文件 |
| 内联注释 | 函数/模块文档、复杂逻辑说明、TODO/FIXME |
| 测试代码 | 单元测试、集成测试、E2E 测试 |
| 构建脚本 | CI/CD、构建工具、部署脚本 |

**维护原则**：代码即文档，注释解释"为什么"而非"是什么"

### 文档维护规则

1. **PLAN.md**：每次大版本更新同步，保留修改摘要
2. **WHITEPAPER.md**：架构变更时更新，保持与代码一致
3. **docs/*.md**：功能开发时创建，完成后保留归档
4. **代码注释**：随代码提交，Review 时检查
